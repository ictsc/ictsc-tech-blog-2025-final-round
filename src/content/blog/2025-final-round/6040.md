## 問題文

あなたは業務システムの開発をしていましたが、「時代はソーシャルゲーム、ビックウェーブだ！」という鶴の一声によりゲーム開発の新規部署でクライアントを開発することになりました。

開発環境の8080番ポートではモックサーバーが動いており開発できそうに見えましたが、本番サーバーに繋ごうとしたところうまくいきません。どうやら既存のインフラに対応させるためgRPC-Webなるものを使っているようです。チームの他の人も困っているので原因を究明して教えてあげましょう。

### 制約

* モックサーバーと本番サーバーの間でAPI仕様は完全に同一です
* 報告書には初期状態で失敗した理由を含めることが強く推奨されます

### 初期状態

* 開発環境から`buf curl --protocol grpc --http2-prior-knowledge http://localhost:8080/game.v1.GameService/GetHomeStatus`でレスポンスが取得できる
* 開発環境から`buf curl --protocol grpc https://server:443/game.v1.GameService/GetHomeStatus`でレスポンスが取得できない

### 終了状態

* 開発環境から`server:443/game.v1.GameService/GetHomeStatus`のレスポンスが取得できる
* 開発環境から`game.v1.GameService`のその他のメソッドのレスポンスが取得できる

### 接続情報

| ホスト名 | IPアドレス | ユーザ | パスワード|
| --------- | ----------- | ------ | ------------------ |
| 開発環境 | 192.168.40.1 | user | ictsc2025 |

## 解説

今回の問題はgRPCとgRPC-Webのプロトコル上の差異に関するものでした。

初期状態で通らないとされているリクエストについて、`-v`オプションを付けて何が起きているのか確認してみます。
```console
$ buf curl -v --protocol grpc https://server:443/game.v1.GameService/GetHomeStatus
buf: * Using server reflection to resolve "game.v1.GameService"
buf: * Dialing (tcp) server:443...
buf: * Connected to 10.0.0.2:443
buf: * ALPN: offering h2,http/1.1
buf: * TLS connection using TLS 1.3 / TLS_AES_256_GCM_SHA384
buf: * ALPN: server accepted h2
buf: * Server certificate:
buf: *   subject: CN=server
buf: *   start date: 2026-03-12 04:46:36 +0000 UTC
buf: *   end date: 2036-03-09 04:46:36 +0000 UTC
buf: *   subjectAltNames: [server, 192.168.40.2]
buf: *   issuer: CN=ictsc-local-ca
buf: * Server certificate chain verified
buf: * Server certificate is valid for server
buf: > (#1) POST /grpc.reflection.v1.ServerReflection/ServerReflectionInfo
buf: > (#1) Accept-Encoding: identity
buf: > (#1) Content-Type: application/grpc
buf: > (#1) Grpc-Accept-Encoding: gzip
buf: > (#1) Te: trailers
buf: > (#1) User-Agent: grpc-go-connect/1.19.1 (go1.26.1) buf/1.66.1
buf: > (#1)
buf: } (#1) [5 bytes data]
buf: } (#1) [21 bytes data]
^CFailure: failed to find service named "game.v1.GameService" in schema
```

`grpc.reflection.v1.ServerReflection/ServerReflectionInfo`でレスポンスが返らずに停止していることがわかります。
このメソッドは[gRPC Server Reflection](https://grpc.io/docs/guides/reflection/)によるものです。
本来gRPCを利用するには双方でスキーマを知る必要がありますが、`buf`や`grpcurl`といったコマンドはこの機能を使ってサーバーからスキーマを取得することで、明示的にスキーマを指定することなくgRPCの利用を可能にしています。
